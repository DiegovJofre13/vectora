# Decisiones de diseño

Registro vivo de decisiones técnicas de Vectora, en orden cronológico por fase. El objetivo es que cualquiera pueda entender *por qué* está construido así sin tener que preguntar.

## Fase 1 — Esqueleto

### Entorno de desarrollo (bloqueante, resuelto)
La máquina no tenía Node, npm, Homebrew ni git instalados, y tampoco las Xcode Command Line Tools (que proveen git en macOS). Sin Homebrew no hay forma de instalar nada vía brew, y git requiere el instalador gráfico de CLT (que pide clic humano) o una contraseña sudo que el agente no puede ingresar.

Se resolvió así:
- **Node.js**: se descargó el binario portable oficial (`node-v22.13.1-darwin-arm64.tar.gz` desde nodejs.org) a `~/.local/tools/node`, sin necesidad de Homebrew ni Xcode CLT (los binarios de Node no requieren compilación nativa). Se debe anteponer `~/.local/tools/node/bin` al `PATH` para usar `node`/`npm`/`npx`.
- **git**: se disparó `xcode-select --install` para que el usuario complete la instalación gráfica en paralelo mientras se avanzaba con el resto del trabajo. `git init` y el primer commit quedan pendientes hasta que la instalación termine.
- El **README** documenta ambos pasos para que cualquier otra máquina en el mismo estado pueda replicar el setup sin fricción.

### Monorepo con npm workspaces (no pnpm/yarn)
Se eligió `npm workspaces` porque ya viene con Node (no depende de instalar un gestor de paquetes adicional, relevante dado el punto anterior) y alcanza para 3 paquetes (`probe`, `server`, `client`). Si el proyecto crece mucho se puede migrar a pnpm sin drama.

### `@vectora/probe`: transporte HTTP local, no invocación in-process
El SDK necesita que Vectora (un proceso separado, potencialmente en otra máquina) pueda invocar la función registrada por el cliente. Se decidió que `probe.register(fn)` levanta automáticamente un servidor HTTP local (`GET /probe/salud`, `POST /probe/ejecutar`) en vez de exponer solo una API in-process.

**Por qué:** reflejar la arquitectura real del producto — el cliente instala el SDK dentro de *su* proceso, que corre en *su* infraestructura, y Vectora le pega por red. Esto es válido para los 3 patrones (A, B, C) por igual: en C el cliente igual expone su función registrada vía el probe aunque internamente llame a su propia API HTTP. Mantiene al SDK con cero dependencias (usa `node:http`, no Express/Fastify) porque es una pieza que vive en infraestructura ajena y debe ser lo más liviana y auditable posible.

El paso "conectar sistema" del Módulo 1 (Fase 2) le pedirá al usuario la URL del probe (`http://localhost:4501` en el caso demo) y hará un `GET /probe/salud` + un `POST /probe/ejecutar` de prueba antes de continuar.

### `wrap` solo mide latencia; el costo se estima server-side
`probe.wrap(ctx, llamadaModelo)` no intenta capturar tokens de entrada/salida, porque la forma de la respuesta del cliente es arbitraria (`string | Record<string, unknown>`) y no se le puede forzar un contrato de métricas de uso sin romper la simplicidad del SDK. En cambio:
- `wrap` mide latencia real (wall-clock) alrededor de la llamada al modelo, que es información que el SDK sí puede observar sin cooperación del cliente.
- El costo se estima en el servidor con una heurística de caracteres (`(chars_entrada + chars_salida) / 4 ≈ tokens`) multiplicada por el precio del catálogo. Es una aproximación razonable para el MVP; **CONNECT-REAL-MODELS.md** (Fase 4) explicará cómo reemplazarla por el conteo de tokens real que devuelven las APIs de los proveedores.

### Catálogo de modelos en archivo, no en DB
`server/src/engine/modelCatalog.ts` es un array estático (5 modelos: 1 frontera, 2 intermedios, 1 barato, 1 open-weights). Es el mismo panel para todas las organizaciones y cambia por despliegue de Vectora, no por cliente — no amerita una tabla. Si a futuro el panel se vuelve configurable por cliente, se migra a DB en ese momento.

### `sugerirModelos(casoDeUso)` aislada como único punto de reemplazo
`server/src/engine/suggestModels.ts` es hoy una heurística basada en regex sobre nombre/descripción (alto riesgo → incluye frontera; alto volumen → baratos + open). Se aisló deliberadamente como la única función que conoce la lógica de sugerencia, para que el día que sea data-driven (entrenada sobre qué modelo ganó históricamente) el reemplazo sea quirúrgico: misma firma, mismo call site, cero cambios en la UI ni en el resto del motor.

### `requiereGeneradorParaTipo` y `estrategiaScoringParaTipo` como fuente única de verdad
`server/src/engine/taskTypes.ts` centraliza la distinción "¿el caso trae su input o hay que generarlo?" y qué estrategia de scoring le corresponde a cada tipo de tarea. Tanto el seed como (en Fase 2) la API de creación de casos de uso llaman a estas funciones — así el criterio no se duplica ni diverge entre el seed y el motor real.

### Motor Mock: la única pieza mockeada
`server/src/engine/mockModelEngine.ts` reemplaza la llamada cara a un modelo real. Simula latencia realista (con jitter ±20% sobre la latencia base del catálogo) y una probabilidad de "acierto" (`calidadBase` del catálogo) que decide si la respuesta sintetizada está bien anclada al contexto o degradada — así el juez y el scoring estructural (Fase 2) tienen señal real que detectar, no un mock plano que siempre "sale bien". Todo lo demás (SDK, ganchos, HTTP) opera de verdad.

### Prisma sin `enum`, para portabilidad a Postgres
El conector `sqlite` de Prisma no soporta `enum` nativo. Como el proyecto está pensado para migrar a Postgres, todos los campos "tipo" (`tipoTarea`, `estado`, `tier` en el catálogo, etc.) son `String` validados en la capa de aplicación, no `enum` de Prisma. Esto hace que el cambio de `datasource` a Postgres sea un cambio de una línea, sin tocar el schema de modelos.

### Nombres de campo de `CorreccionJuicio`
El contrato pedido especifica campos en inglés (`useCase, domain, question, context, systemAnswer, provisionalExpected, judgeVerdict, humanVerdict, confidence, timestamp`). El resto del schema está en español (consistente con el resto del producto). Se mapeó 1:1: `useCase→casoDeUsoId` (FK, en vez de duplicar el string), `domain→dominio`, y el resto se dejó tal cual porque son datos de texto libre sin equivalente de dominio claro en español que no sea una traducción literal. Se agregó `correctedAnswer` (no pedido explícitamente) para separar "corrigió el texto" de solo marcar `humanVerdict = "corregida"` sin perder el texto corregido — necesario para que el dato sirva a fine-tuning futuro.

### Seed "Fintech Andina": generado sin pasar por el motor Mock en vivo
El seed (`server/prisma/seed.ts` + `seedData.ts`) sintetiza directamente filas de Prisma con un RNG de semilla fija (`mulberry32`), en vez de invocar el motor Mock real (que usa `setTimeout` reales para simular latencia). Poblar ~275 `ResultadoModelo` + 200 `CorreccionJuicio` a través de llamadas HTTP reales habría tomado minutos y sido no-determinista; el seed necesita ser instantáneo y reproducible. La lógica de síntesis de scores/latencia/costo en `seedData.ts` es deliberadamente independiente del motor Mock real — están calibradas para verse consistentes, pero no comparten código, porque uno sirve para *demostrar* datos históricos plausibles y el otro para *ejecutar* evaluaciones nuevas de verdad.

Volumen sembrado: 5 casos de uso, 5 corridas de evaluación, 55 casos de prueba, 275 resultados por modelo, 200 correcciones de juicio calibradas, 6 resultados con confianza < 0.65 (cola pendiente de calibración del Módulo 3), 4 eventos de gobernanza, 5 movimientos de crédito.

### Fixtures demo como sistemas reales, no simulados en el motor
`server/src/demo/fintechAndinaBot.ts` (patrón A, RAG) y `fintechAndinaFraude.ts` (patrón B, clasificación) son programas reales que usan `@vectora/probe` de verdad — se levantan como procesos propios (`npm run dev:demo-bot` / `dev:demo-fraude`) y exponen su servidor HTTP en los puertos 4501/4502. Esto prueba el contrato end-to-end (register → wrap → intercambio de modelo → HTTP) sin atajos, y le da al onboarding un ejemplo real y corriendo contra el cual apuntar el paso "conectar sistema" en Fase 2.

### Tailwind v3 (no v4) para el client
Se usó la configuración clásica de Tailwind v3 (`tailwind.config.js` + PostCSS) en vez de v4 (que elimina el archivo de config a favor de CSS-first). La razón es pragmática: v3 permite declarar la paleta de marca completa (`fondo`, `superficie`, `linea`, `tinta`, `marca`, `ambar`, `coral`, `azul`, `violeta`) como tokens de Tailwind de forma más explícita y documentada, y es la versión con la que hay más certeza de compatibilidad estable hoy.

### Vulnerabilidad conocida y aceptada: esbuild/vite (dev-only)
`npm audit` reporta una vulnerabilidad moderada en `esbuild <=0.24.2` (usado por Vite 5): un sitio malicioso podría enviar requests al dev server y leer la respuesta. Solo afecta al servidor de desarrollo local (no a builds de producción) y arreglarla requiere saltar a Vite 8 (breaking change no evaluado). Se deja pendiente para revisar en Fase 4 (pulido), no se ignora silenciosamente.

## Fase 2 — Conectar y evaluar (Módulos 1 y 2)

### El KB / documentos existentes se suben directo a Vectora, no a través del probe
El contrato del SDK (`register` + `wrap`) es deliberadamente mínimo: no hay un tercer gancho para "listar documentos" o "exponer el knowledge base". Pero el agente generador necesita leer contenido para producir preguntas, y el scoring estructural necesita un ground truth (RUT/monto/fecha correctos) contra el cual comparar. Se decidió que el paso 2 del stepper le pide estos insumos **directamente al usuario en la UI de Vectora** (textarea de documentos KB, o pares documento+campos-esperados para extracción/clasificación), no a través del probe. Esto es consistente con "el cliente da acceso completo a su sistema... incluido el knowledge base cuando el caso lo requiera" — es acceso separado del gancho de ejecución, no otro hook del SDK. `POST /api/casos-de-uso/:id/evaluaciones` recibe `kbDocs` o `documentosExistentes` junto con los modelos elegidos, en la misma llamada que dispara la corrida.

### El agente generador es plantillas deterministas, no una llamada LLM real
Generar preguntas realmente novedosas a partir de texto arbitrario requeriría una llamada de generación a un LLM — que no existe en el motor Mock (solo se mockea la llamada del *cliente*, no hay un "modelo generador" real disponible). `generatorAgent.ts` sintetiza preguntas con plantillas que combinan el título del documento con una frase real extraída de su contenido (no solo el título), para que la pregunta comparta vocabulario genuino con la respuesta esperada. Sin esto, el juez léxico (ver abajo) no tendría ninguna señal real que medir. Documentado como aproximación explícita, reemplazable por una llamada real más adelante.

### El juez es una heurística de solapamiento léxico real, no números fabricados
A diferencia del seed de Fase 1 (que fabricaba scores directamente para verse bien en la demo), el juez de la Fase 2 (`engine/judge.ts`) **calcula de verdad** groundedness/relevancia/completitud comparando conjuntos de palabras entre pregunta, contexto recuperado, respuesta y referencia provisional — sin llamar a ningún modelo. Es honesto: los scores resultantes son modestos (~45-60% en las pruebas end-to-end) porque un solapamiento léxico crudo es un proxy imperfecto de calidad real, no un juez semántico. Se prefirió esto a inflar los números artificialmente — un run real contra el motor Mock debía producir un resultado *computado*, no actuado. La confianza del juez se deriva de cuánta varianza hay entre los 3 criterios (más varianza → menos confianza), lo que además genera de forma orgánica los casos de baja confianza que alimentan el Módulo 3, sin tener que forzarlos.

### Scoring estructural: comparación exacta salvo campos declarados ambiguos
`engine/structuralScoring.ts` compara campo por campo contra el ground truth que el usuario aportó. Los campos marcados como `camposAmbiguos` (ej. "resumen", "motivo") usan similitud de palabras en vez de igualdad exacta — el resto exige coincidencia exacta normalizada (minúsculas, sin acentos). Sin juez LLM, tal como pide la arquitectura de scoring.

### Orquestador: fire-and-forget + polling, no websockets
`POST .../evaluaciones` crea la corrida y sus casos de prueba sincrónicamente (rápido, sin llamadas de red), y dispara la ejecución real en segundo plano sin esperarla (`void ejecutarCorridaEnSegundoPlano(...)`). La UI hace polling a `GET .../progreso` cada 1.2s. Se prefirió esto a WebSockets/SSE porque el progreso es una consulta de agregación simple sobre `ResultadoModelo` (cuenta cuántos ya existen por modelo) — no hay estado en memoria que sincronizar, así que un restart del server no deja el progreso "colgado": la próxima consulta simplemente refleja lo que ya está en la DB.

### Rate limiting: límite de concurrencia propio, no una librería
`lib/pLimit.ts` es un limitador de concurrencia de ~20 líneas (cola + contador de tareas activas), usado para no disparar los ~150 POST al probe del cliente todos a la vez (`CONCURRENCIA_MAXIMA = 4` en `orchestrator.ts`). No se agregó `p-limit` como dependencia por algo tan simple.

### Ranking del reporte: "ajuste al caso de uso", no precisión pura
`engine/report.ts` ordena la tabla por `ajusteCasoUso = retencionCalidad·0.7 + (1 − costoRelativo)·0.3`, donde `retencionCalidad` es la precisión del modelo relativa a la más alta del panel. Esto es lo que permite recomendar "el barato-suficiente" en vez de siempre al más preciso — el modelo con mayor `ajusteCasoUso` es el veredicto. Los tags (óptimo/valor/máxima precisión/open) se asignan por prioridad: óptimo (el recomendado) > máxima precisión (score más alto) > open (openWeights) > valor (está en la frontera de Pareto pero no es ninguno de los anteriores).

### Frontend: estado de vista en React, sin librería de routing
Con 3 vistas (lista, stepper de nuevo caso, reporte) un `useState<Vista>` en `App.tsx` alcanza — no se agregó `react-router` para esto. Si el número de vistas crece (ej. detalle de caso con URL propia, deep-linking a un reporte), se reconsidera.

### Verificación visual con Playwright (herramienta ad hoc, no dependencia del repo)
No había navegador disponible en el entorno por defecto. Se instaló Chromium vía `npx playwright install chromium` y se corrió un script de verificación end-to-end (crear caso → conectar al probe demo → cargar KB → correr evaluación real → ver reporte) desde un directorio de scratch, fuera del repo. No se agregó `playwright` como dependencia del proyecto — fue una herramienta de verificación puntual para esta sesión, no parte del producto. Encontró y permitió corregir dos problemas reales: (1) los porcentajes de precisión se mostraban redondeados a enteros, lo que hacía que modelos con scores genuinamente distintos (48.7%, 48.9%, 49.1%) se vieran idénticos — se cambió a un decimal; (2) las etiquetas de puntos cercanos en el gráfico de Pareto se superponían — se alternan arriba/abajo según orden de costo.

## Fase 3 — El foso (Módulos 3 y 4)

### `CorreccionJuicio.resultadoModeloId`: se agregó un FK que el contrato original no pedía
El contrato de `CorreccionJuicio` (Fase 1) no incluía un vínculo al resultado puntual que la originó — tenía sentido para el seed histórico (correcciones fabricadas, no ligadas a una corrida en particular), pero el Módulo 3 en vivo necesita saber **qué resultados de baja confianza ya fueron calibrados** para no mostrarlos de nuevo en la cola. Se agregó `resultadoModeloId String? @unique` (nullable porque las 200 correcciones sembradas en Fase 1 no tienen un resultado vivo detrás). La cola de pendientes (`obtenerPendientesCalibracion`) filtra por `confianzaJuez < 0.65 AND correccionJuicio: null` — así calibrar un ítem lo saca de la cola de inmediato, sin necesitar un campo de estado adicional en `ResultadoModelo`.

### El seed estructural se corrigió para ser re-corrible de verdad
Los `CasoPrueba.input` de los casos estructurales sembrados en Fase 1 (`{id, resumen}`) no calzaban con el sobre `{documento, esperado, camposAmbiguos}` que espera `orchestrator.ts` para scoring estructural en vivo. Mientras solo se leían los scores ya fabricados por el seed, esto no importaba — pero el Módulo 4 necesita poder **re-correr** esos casos de prueba guardados para simular "salió un modelo nuevo", y con el sobre viejo el scoring en vivo habría dado 0 en todos los modelos (sin `esperado`, degenerado). Se corrigió el seed para usar el mismo sobre que el motor real, así cualquier caso con probe conectado es genuinamente re-corrible, no solo los RAG.

### "Simular modelo nuevo" re-ejecuta el sistema real, no anima un resultado
`engine/governance.ts::simularEventoNuevoModelo` no fabrica un veredicto: toma un modelo del catálogo que el caso todavía no había evaluado, copia los casos de prueba guardados (las mismas ~30 preguntas o documentos), crea una `EvaluacionCorrida` nueva con el panel anterior + el modelo nuevo, y la corre de verdad contra el probe del cliente (reusando `ejecutarCorridaParaGobernanza`, la misma función que usa una corrida normal, solo que aquí se espera su resultado en vez de dispararla fire-and-forget). Compara el `modeloRecomendado` de antes y de después — si cambia, `huboImpacto = true`. Una corrida de 30×5 casos tarda ~45-90s reales; el botón queda deshabilitado y en estado "Simulando…" mientras corre. No hay atajo: si el modelo nuevo no cambia nada, el evento se registra igual (huboImpacto=false) porque el historial debe incluir "eventos sin cambio".

**Limitación conocida**: solo es simulable para casos con `probeUrl` conectado (Bot de soporte y Detección de fraude, de los 5 sembrados) — los otros 3 son entradas de ledger sin sistema real detrás, y el botón queda deshabilitado con un tooltip que lo explica, en vez de fingir una simulación. El seed deja a "Bot de soporte" con 4 de los 5 modelos del catálogo evaluados (sin `llama-3-1-70b`) a propósito, para que la demo tenga un modelo "nuevo" real que ofrecer sin agotar el catálogo en el primer intento.

### Estado de gobernanza (óptimo / cambio sugerido / evaluación vieja) es derivado, no persistido
No hay un campo `estadoGobernanza` en `CasoDeUso` — se calcula al vuelo en `calcularFilaLedger()` a partir de: (a) si pasaron más de 60 días desde la última evaluación completada → "evaluación vieja"; (b) si no, y el modelo recomendado del último reporte difiere del `modeloProduccion` declarado → "cambio sugerido"; (c) si no → "óptimo". Evita que este estado se desincronice del reporte real (que si estuviera persistido, requeriría recalcularse en cada evento) a costa de recalcular el reporte de la última corrida en cada carga del ledger — aceptable dado el volumen (≤10s casos de uso en el MVP).

### Tarjetas resumen: aproximaciones explícitas, no contabilidad real
"Gasto mensual de IA" suma `costoMensualProduccion` declarado por caso (dato que el usuario ingresa, no medido). "Ahorro acumulado" multiplica ese costo por el `ahorroPctVsProduccion` del último reporte, solo cuando el modelo en producción fue parte del panel evaluado (si no, no hay base de comparación y no suma nada). Es una estimación de negocio, no un ledger contable — coherente con que todo el módulo de créditos/costos del MVP es de pago-por-uso simulado, sin procesar pagos reales.

### Formato de contexto legible en la cola de calibración
El contexto recuperado se guarda como JSON (`[{id, titulo, contenido}]`) para que el reporte pueda re-derivarlo estructuradamente. Mostrarlo tal cual en el Módulo 3 (JSON crudo) fue el primer intento y se veía mal en la revisión visual — `calibration.ts::textoLegible()` ahora detecta ese shape específico y lo aplana a texto (`titulo: contenido`, unido con saltos de línea) antes de mandarlo a la UI.

### Verificación visual encontró un bug real: POST sin body + `content-type: application/json`
El botón "Simular modelo nuevo" no mandaba body (no lo necesita), pero el cliente HTTP (`lib/api.ts::req`) siempre setea `content-type: application/json`. Fastify rechaza esa combinación (`FST_ERR_CTP_EMPTY_JSON_BODY`) — el botón fallaba en silencio en el primer clic real desde el navegador (curl con `-d` sí manda body, por eso no apareció en las pruebas de backend). Se arregló mandando `body: "{}"`. Un recordatorio de por qué la verificación en navegador real importa más que solo probar el backend con curl.

## Fase 4 — Pulido

### `CONNECT-REAL-MODELS.md`: el punto de contacto real está fuera de Vectora
Documentar "cómo pasar a modelos reales" obligó a ser explícito sobre algo que el diseño ya daba por sentado: Vectora **nunca** llama a un modelo. La única llamada real vive dentro del sistema del cliente, en el callback de `probe.wrap`. Para el caso común (un cliente conectando su propio sistema), no hay ningún cambio que hacer en el código de Vectora — el documento tuvo que aclarar esto primero, antes de entrar en los cambios opcionales (fixtures demo con modelos reales, juez LLM real, generador con LLM real, costos con uso real de tokens). Ver `CONNECT-REAL-MODELS.md`.

### Export PDF: `window.print()` + CSS de impresión, no un renderer server-side
Se consideró generar el PDF en el server (ej. con Puppeteer) para tener control total del layout, pero eso agrega una dependencia pesada (un Chromium embebido) solo para un botón. `window.print()` del navegador **ya produce un PDF real** cuando el usuario elige "Guardar como PDF" — el trabajo real fue agregar una hoja de estilos `@media print` (clase `print:hidden` de Tailwind en el header/nav/botones, un encabezado propio solo-para-impresión con marca y fecha ya que el header normal se oculta, `break-inside: avoid` en tarjetas y tabla para que no se corten a la mitad entre páginas). Verificado con `page.pdf()` de Playwright: genera un PDF real de ~300KB con el layout esperado.

### Estados vacíos y de error: auditoría sistemática, no solo el caso feliz
Se revisaron las 6 páginas del client buscando specifically **promesas sin `.catch`** (quedan como "cargando" para siempre si la request falla) y **colecciones sin mensaje cuando vienen vacías** (se renderiza una tabla con solo encabezados, sin explicar por qué no hay filas). Se encontraron y corrigieron 3 casos reales: el catálogo de modelos y el costo inicial en el stepper (Paso 2 y 3) no capturaban errores de red, y la tabla de gobernanza no tenía mensaje para cuando no hay casos de uso todavía. En el server se agregaron chequeos de existencia antes de escribir (`verificar-conexión` sobre un caso inexistente, `organizacionId` inválido al crear un caso) en vez de dejar que Prisma tire un error genérico sin capturar.

### Onboarding realmente vacío: se probó de verdad, vaciando la base
El spec pide "un cliente nuevo puede instalar el probe, declarar sus ganchos, conectar su primer caso y correr una evaluación" — sin datos sembrados. Encontramos que faltaba una pieza real: no existía forma de crear una `Organizacion` desde la UI, así que un deploy sin seed se habría quedado bloqueado en el paso 1 del stepper ("No hay ninguna organización configurada"). Se agregó `POST /api/organizaciones` y el stepper ahora auto-provisiona "Mi organización" la primera vez que no encuentra ninguna. Esto se verificó vaciando la base de datos por completo (no editando código para simularlo) y recorriendo lista vacía → gobernanza vacía → calibración vacía → crear el primer caso de uso desde cero.

### El gráfico de Pareto necesitó una segunda pasada de colisión de labels
La verificación visual de la Fase 2 ya había arreglado una superposición de labels (alternando arriba/abajo por orden de costo). En la Fase 4, con un panel de 4 modelos reales, apareció una colisión distinta: el label de un modelo barato con precisión cercana a una marca del eje (25/50/75/100%) quedaba encima del número del eje — el algoritmo de colisión conocía las posiciones de los demás labels de puntos, pero no las de los ticks del eje Y. Se corrigió sembrando el set de "posiciones ya ocupadas" con los 5 ticks del eje antes de colocar los labels de los puntos, reusando el mismo mecanismo de colisión en vez de agregar un caso especial. Confirma el patrón general: en un componente con pocos elementos (≤5 puntos), colisión por fuerza bruta contra todo lo que ya está en el lienzo es más robusto que alternar por una sola heurística (orden de costo, en este caso) que solo cubre un tipo de choque.

## Fase 5 — Reducir el foco: probar conexión real + ver el detalle por caso

El resto del roadmap (foso, ledger, alertas) quedó en pausa a propósito para validar primero las dos cosas que sostienen la hipótesis de valor: que la conexión a un sistema real funcione de verdad, y que el usuario pueda confiar en una evaluación viendo el detalle caso por caso, no solo el agregado.

### Se encontró y corrigió un problema real antes de documentarlo: sin timeout
Al releer `orchestrator.ts` para escribir `docs/COMO-FUNCIONA-LA-CONEXION.md` (que pide ser honesto sobre limitaciones), apareció un hallazgo que no era solo para documentar: ninguna llamada al probe del cliente tenía timeout — un sistema real lento o colgado ocupaba para siempre uno de los 4 cupos de concurrencia y la corrida nunca terminaba, sin ningún mensaje de error. Se agregó `AbortController` con 60s en `orchestrator.ts::llamarProbe` (la llamada real, dentro de una corrida) y 10s en `casosDeUso.ts::fetchConTimeout` (la verificación de conexión, que debe ser rápida). Documentar sin arreglar algo tan barato y tan directamente relevante para "probar con sistemas reales esta semana" no tenía sentido.

### `docs/COMO-FUNCIONA-LA-CONEXION.md` se escribió releyendo el código, no la spec
El pedido explícito era describir el sistema tal como quedó construido. Esto sacó a la luz cosas que no estaban documentadas en ningún lado: que no hay autenticación entre Vectora y el probe del cliente (HTTP plano, sin API key — aceptable en localhost, no para exponer a internet sin túnel), que `verificar-conexión` prueba siempre con `modelo: "gpt-4o-mini"` y un input string fijo (puede dar falso negativo en extracción/clasificación, donde se espera un objeto), y que el SDK no tiene forma de detectar si el cliente ignora el parámetro `modelo` — el síntoma visible es que el reporte muestra a todos los modelos con métricas casi idénticas. Estas tres cosas quedaron en la sección de Limitaciones conocidas, en vez de callarlas.

### `examples/cliente-demo/`: workspace propio, no otro fixture dentro de `/server`
Los fixtures de Fintech Andina (`server/src/demo/*`) importan del motor Mock del propio server — sirven para la demo interna, pero no son un buen ejemplo de "cómo conecto mi sistema" porque un cliente real no tiene acceso a `server/src/engine/mockModelEngine.ts`. `examples/cliente-demo/` es un workspace nuevo e independiente: solo depende de `@vectora/probe` (como lo haría un cliente real), lee su propio knowledge base de `kb/*.md` en disco, y su único archivo con lógica de proveedor (`src/llm.ts`) ya sabe llamar a la API real de OpenAI si hay `OPENAI_API_KEY` en el entorno — sin esa key, cae a un stub local para que el ejemplo funcione sin configuración. Verificado de punta a punta: `npm run dev:cliente-demo` → `verificar-conexión` desde el server de Vectora real → `200 ok`.

### `CorreccionJuicio` y `CasoPrueba.contextoFuente` no alcanzaban para el detalle por caso
El reporte agregado (Fase 2) no necesitaba guardar el *por qué* de cada score, solo el número. Mostrar el detalle por caso sí lo necesita, así que `ResultadoModelo` ganó 3 campos: `veredictoJuez` ("paso"|"fallo", derivado de `scorePromedio` vs. un umbral de 0.55 — documentado como ajustable, no una cifra mágica), `razonamientoJuez` (texto compuesto a partir de los mismos números que ya calculaba el juez, banda por banda — alto/medio/bajo — no un texto aparte inventado), y `detalleEstructural` (JSON con el desglose campo por campo para extracción/clasificación, mismo patrón: veredicto a partir de un umbral, esta vez 0.75).

Separado de eso, `CasoPrueba.contextoFuente` pasó de guardar solo los ids de los documentos del KB (`"kb-devoluciones,kb-limites"`) a guardar los documentos completos (`[{id, titulo, contenido}]`). Sin este cambio, la trazabilidad que pedía el Entregable 2 ("quiero ver que la pregunta salió del contenido real") no se podía mostrar para casos creados en vivo: los documentos del KB que sube un usuario por la UI nunca se guardaban en ningún lado más que en la memoria de la corrida — solo existían los ids, sin nada a qué apuntar. `generatorAgent.ts::generarPreguntas()` ahora devuelve `documentosFuente` (los documentos reales, no solo sus ids) en cada `PreguntaGenerada`, y el seed se actualizó al mismo formato para que los datos sembrados y los datos en vivo se vean iguales en la UI.

### El score estructural del seed generaba porcentajes fraccionarios en campos que no son ambiguos — bug real, encontrado en la revisión visual
El seed original sorteaba un score continuo por tier (ej. 0.87) y lo repartía entre 2 campos ficticios para que sumaran ese número. El problema, invisible hasta que el Entregable 2 expuso el detalle campo por campo: un campo no declarado como ambiguo solo puede coincidir o no coincidir (puntaje 0 o 1) en el motor real (`structuralScoring.ts`) — mostrar "campo2: ~91% valor-extraido-2" para un campo exacto no ambiguo es una combinación que el motor real nunca produce, y quedaba mal en la UI (parecía un score de similitud donde no lo hay). Se corrigió `seedData.ts::sintetizarCamposEstructural()` para sortear cada campo de forma binaria (según la probabilidad de acierto del tier) y derivar el score agregado del promedio — igual que haría el motor real. Efecto secundario aceptado: para los 4 casos sembrados con scoring estructural, el score por caso puntual ahora es más "escalonado" (0%, 50%, 100% con 2 campos) en vez de continuo; el score agregado del reporte (promedio sobre 5-10 casos) sigue viéndose razonable porque promedia muchos casos.

### La vista de detalle es una sola pantalla con acordeón, no dos pantallas separadas
El pedido describe "2a: set de pruebas generado" y "2b: panel de detalle por caso" como dos cosas. Se implementaron como una sola vista (`DetalleCasos.tsx`): cada caso es una fila colapsada que ya muestra lo de 2a (pregunta, dificultad, trazabilidad al documento fuente) sin necesitar abrirla, y al hacer clic se expande al detalle completo lado a lado por modelo (2b). Evita construir navegación entre dos pantallas para ver información que de todos modos hay que correlacionar caso por caso, y cumple el pedido explícito de "poder recorrer los 30 casos rápido": los indicadores de "falló" / "baja confianza" / "discrepancia" se calculan y muestran en la fila colapsada, así que escanear la lista ya dice dónde mirar antes de expandir nada.

### Accesible durante la corrida, no solo al terminar
Los `CasoPrueba` se crean síncronamente al iniciar la corrida, antes de llamar al probe del cliente ni una vez (`orchestrator.ts::iniciarCorrida`) — así que `GET .../casos` siempre devuelve algo coherente, con `resultados` incompleto mientras la corrida sigue corriendo. `PasoCorrer.tsx` expone un link "Ver el set de casos generado" apenas existe un `corridaId`, sin esperar a que termine.

### Verificación visual encontró 2 bugs reales, otra vez
`resumirInput()` no manejaba documentos anidados (`{documento: {id, resumen}}`, la forma que usa el seed) y mostraba JSON crudo como título de la fila — se corrigió con una función recursiva de 2 niveles. Y el bug de scoring estructural fraccionario de arriba. Ninguno de los dos aparece en un chequeo de tipos ni en una prueba de backend con curl — ambos solo se ven mirando la pantalla real.

## Fase 6 — Gateway de modelos: Vectora paga, cobra créditos con margen

Cambio de arquitectura real, no un ajuste menor: hasta acá, Vectora nunca llamaba a un modelo — la llamada real vivía siempre dentro del sistema del cliente, con su propia key. Ahora existe un segundo camino, complementario (no reemplaza al primero): el cliente puede pedirle a Vectora que haga la llamada real (solo OpenAI por ahora, confirmado con el negocio), Vectora paga con su propia cuenta, y le cobra al cliente créditos = costo real + margen (30%, confirmado con el negocio). Es el modelo de negocio "pago por uso" ya descrito en el spec original, ahora con dinero de verdad de por medio en vez de un contador simbólico.

### La key de OpenAI del negocio se recibió por chat — se guardó de inmediato, nunca se commiteó
El usuario pegó una API key real de OpenAI directo en la conversación. Antes de construir nada, se guardó en `server/.env` (ya gitignorado, confirmado antes de escribir) y no se volvió a imprimir en ningún output de terminal ni commit. Vale decirlo explícito: **cualquier secreto que llegue por chat queda expuesto en ese historial de conversación**, que es un canal fuera del control de este repo — la recomendación al usuario fue rotar la key después de las pruebas si le preocupa, no se hizo nada más allá de guardarla con cuidado acá.

### `probe.completar()` es aditivo, no reemplaza `wrap()` a nivel de SDK — pero sí a nivel de qué se muestra
Se pudo haber cambiado el contrato de `wrap` para que opcionalmente delegue al gateway, pero eso arriesga romper toda integración BYO-key ya existente (los 3 patrones documentados, el ejemplo, los fixtures demo). En cambio, `probe.completar(ctx, { prompt }): Promise<{ texto }>` es un método nuevo y paralelo — a nivel de código, `wrap` sigue existiendo intacto. *(Nota: la idea original acá era que "el cliente elige cuál usar por sistema, y ambos pueden convivir" — eso se revirtió a propósito como política de producto, ver Fase 7 más abajo. `wrap` sigue en el SDK por compatibilidad interna, pero ya no se muestra como opción.)*

### El costo se calcula con tokens reales de la API, no con la heurística de caracteres
`estimarCostoUsd` (motor Mock) y `estimarCostoCorrida` (pre-corrida) siguen siendo heurísticas de `chars/4` — están bien para comparar modelos entre sí en el motor Mock, pero acá se cobra plata real, así que `providerGateway.ts` usa `usage.prompt_tokens`/`usage.completion_tokens` que devuelve la propia API de OpenAI, multiplicados por una tabla de precios real por modelo (`PRECIOS_OPENAI`, hardcodeada — hay que actualizarla a mano si OpenAI cambia precios, no hay integración con ninguna API de pricing).

### 1 crédito = 1 USD, a propósito
Se consideró un sistema de "créditos" como unidad abstracta con una tasa de cambio a dólares, pero no aporta nada en el MVP — solo una capa de indirección para explicar después. `Organizacion.saldoCreditosUsd` es literalmente dólares.

### El ledger de "1 crédito por corrida" quedó obsoleto y se eliminó, no se dejó en paralelo
`orchestrator.ts` tenía un `movimientoCreditos.create({ creditosConsumidos: 1, costoUsd: costoRealUsd })` al final de cada corrida — vestigio de cuando "créditos" era un contador simbólico. Con cobro real por llamada dentro del gateway (`routes/gateway.ts::registrarConsumo`, en tiempo real, solo cuando el cliente de verdad usa `completar()`), dejar ese registro de fin de corrida habría **cobrado dos veces** por las llamadas que sí pasaron por el gateway, y habría cobrado **de más** a quien usa su propia key (BYO), a quien no le cuesta nada a Vectora. Se sacó por completo — `costoRealUsd` en `EvaluacionCorrida` queda como referencia informativa para el reporte, ya no dispara ningún cargo.

### El pre-check de saldo en `iniciarCorrida()` es una estimación conservadora, no un cargo exacto
Vectora no puede saber, al recibir la corrida, si el sistema del cliente va a usar `wrap` (su key) o `completar` (el gateway) — esa decisión vive adentro del código del cliente, invisible desde afuera. Se optó por bloquear la corrida completa si el saldo no alcanza para la estimación existente (`costEstimator.ts`) + margen, asumiendo el caso conservador (que todo pasa por el gateway). Es un gate grueso a propósito: el gate fino y exacto, que solo cobra por lo que de verdad ocurre, vive en `routes/gateway.ts`. Mismo chequeo se agregó a `governance.ts::simularEventoNuevoModelo`, que crea su propia `EvaluacionCorrida` por fuera de `iniciarCorrida()` y se había quedado sin este gate.

### Migración de `MovimientoCreditos` sin perder datos existentes
Rediseñar la tabla (de `creditosConsumidos: Int` a `tipo`/`montoUsd`/`costoBaseUsd`/`margenUsd`) con filas existentes rompía `prisma db push` (columnas requeridas sin default, con datos ya cargados). Como una de esas filas pertenecía a la organización real del usuario ("Mi organización", de sus propias pruebas — no a Fintech Andina, que se re-siembra sola), no era aceptable un `--force-reset`. Se respaldaron las 6 filas a un archivo temporal, se vació la tabla, se aplicó el schema nuevo, y se reinsertaron mapeadas a la forma nueva (`tipo: "consumo"`, `margenUsd: 0` para marcar que son anteriores al margen). Cero filas perdidas.

### `apiKeyGateway` es nullable, se genera perezosamente
Igual que con `MovimientoCreditos`, agregar un campo `@unique` no nulo a `Organizacion` habría roto las organizaciones ya existentes (otra vez, la del usuario). Quedó `String?`, y `credits.ts::obtenerOCrearApiKeyGateway()` la genera y persiste la primera vez que se necesita — ninguna organización se queda nunca sin key, pero tampoco hizo falta una migración de datos para esto.

### Verificado con dinero real, no simulado — y con el mínimo gasto posible
Se probó el gateway con llamadas reales y baratas a `gpt-4o-mini` (prompts de una frase, ~$0.0001 gastado en total, visible en el ledger), el bloqueo por saldo insuficiente con una organización nueva en cero (sin gastar nada — la llamada real ni se dispara), la carga de créditos, y el flujo completo del SDK (`cliente-demo` con `VECTORA_API_KEY`, un `POST /probe/ejecutar` de verdad, no los 30 casos). A propósito no se corrió una evaluación completa de 30 casos contra el gateway real en esta sesión — eso ya cuesta una cantidad no trivial de dinero real de la cuenta del negocio, y es una decisión que le corresponde al usuario tomar, no algo para gastar sin más por defecto.

## Fase 7 — Sacar el BYO-key de todo lo que se muestra: el gateway es el único camino soportado

El usuario preguntó explícitamente por qué la documentación mostraba `completar()` como "el cliente elige" entre su propia key o el gateway de Vectora, y fue tajante: **solo debería ser posible usar las de Vectora**. La pregunta obligó a distinguir dos cosas que se habían mezclado: qué puede hacer el código del cliente (nadie puede impedirlo — `fn`, la función que el cliente registra con `probe.register(fn)`, es una caja negra para Vectora, que solo la invoca por HTTP) y qué le muestra y sugiere Vectora al cliente (eso sí es 100% decisión de producto). Se optó por lo segundo: sacar el BYO-key de absolutamente todo lo visible — snippets del stepper (`SnippetProbe.tsx`), el texto de `PasoConectarModelos.tsx`, `examples/cliente-demo` (que antes tenía una rama "con tu propia key" en `llm.ts`, ahora borrada), y los 3 documentos (`docs/COMO-FUNCIONA-LA-CONEXION.md`, `docs/CONECTAR-SISTEMA-REAL.md`, `CONNECT-REAL-MODELS.md`). En todos lados donde antes se explicaba `wrap()` como alternativa con key propia, ahora se dice explícitamente que **no es un camino soportado ni documentado**, y que `completar()` es el único camino.

Se fue honesto sobre el límite: esto es una política de producto, no una garantía técnica. Nada impide que un cliente, dentro de su función registrada, siga llamando a OpenAI directo con su propia key — Vectora nunca ve ni puede inspeccionar ese código. Lo único que cambió es que Vectora dejó de mostrárselo, sugerírselo, o darle un ejemplo copiable para hacerlo. Documentar esa limitación explícitamente (en vez de insinuar una garantía que no existe) es más valioso que aparentar un control que no se tiene.

`wrap()` no se borró del SDK: los fixtures demo de Fintech Andina (motor Mock, sin costo real) lo siguen usando internamente, y romperlo habría sido una regresión sin ningún beneficio para el objetivo de este cambio. Es legacy/interno, no una segunda opción para clientes reales.

Se agregó de paso soporte de `formato: "json"` en `completar()` (necesario para el Patrón B — extracción/clasificación — que antes solo se podía resolver pidiendo JSON dentro del callback de `wrap`), pasado hasta `providerGateway.ts` como `response_format: { type: "json_object" }` de OpenAI.

## Fase 8 — Dos bugs reales encontrados por el usuario probando como cliente

El usuario corrió el flujo completo simulando ser cliente (no una prueba automatizada) y encontró dos bugs reales:

### Bug: el paso 2 del stepper se vacía al volver desde el paso 3
`client/src/pages/NuevoCaso.tsx` renderiza `PasoConectarModelos` condicionalmente (`{paso === 2 && ...}`), así que al avanzar a paso 3 el componente se **desmonta** por completo, y al volver se **remonta** desde cero — con sus `useState` en los valores default de siempre (URL de ejemplo, sin verificar, sin KB, sin modelos elegidos). `NuevoCaso` sí guardaba el resultado del paso (`datosConexion`), pero solo lo pasaba hacia adelante (a `PasoCorrer`), nunca hacia atrás como valor inicial de `PasoConectarModelos`. Se arregló pasándolo como prop `valorInicial`, y rehidratando cada estado local (`probeUrl`, `verificado`, `nombreSistema`, `kbDocs`, `docsExistentes` — con una función `hidratarDocsExistentes` que revierte la transformación a `esperado: Record<string,unknown>`, y `seleccionados`) desde ese valor cuando existe, en vez de siempre arrancar en blanco. También se cuidó que el efecto que auto-selecciona los modelos "sugeridos" no pise esa restauración (`if (valorInicial) return`).

**Nota para más adelante, no arreglada ahora:** `PasoCasoDeUso` (paso 1) tiene la misma forma de bug — si el usuario vuelve del paso 2 al paso 1 y reenvía el formulario, `handlePaso1` en `NuevoCaso.tsx` llama `crearCasoDeUso` de nuevo, creando un `CasoDeUso` duplicado en la base en vez de actualizar el existente (no hay endpoint `PATCH`). No se reportó como bug todavía, pero es la misma clase de problema — si vuelve a aparecer, la solución es la misma: rehidratar `PasoCasoDeUso` desde el `caso` ya creado, y agregar un `PATCH /api/casos-de-uso/:id` en vez de crear de nuevo.

### Bug: el panel dejaba elegir modelos que el gateway no puede llamar
`CATALOGO_MODELOS` (`server/src/engine/modelCatalog.ts`) siempre tuvo 5 modelos de distintos proveedores, pero el gateway real (`providerGateway.ts`) solo sabe llamar a los 2 de OpenAI — `modeloSoportadoPorGateway()` existía pero no se usaba en ningún lado. El resultado: se podía armar y correr una evaluación completa con Gemini/Claude/Llama, y cada caso fallaba recién al llegar al gateway (`El gateway de Vectora todavía no soporta el modelo...`), después de haber gastado tiempo (no crédito — el chequeo de precio pasa antes que cualquier llamada real) configurando todo. Se agregó `gatewaySoportado: boolean` a la respuesta de `/api/catalogo-modelos` (calculado con la función que ya existía), la UI deshabilita esas opciones con una etiqueta "próximamente" en vez de ocultarlas del todo (se van a poder usar cuando se agreguen esas keys, ver ideas para después), y como defensa en profundidad `POST /api/casos-de-uso/:id/evaluaciones` ahora rechaza con 400 cualquier corrida que igual llegue con un modelo no soportado.

Verificado con Playwright en navegador real (no solo compilación): se completó el paso 2 con conexión real a `cliente-demo`, KB de ejemplo y 2 modelos, se avanzó a paso 3, se volvió a paso 2, y se confirmó que la URL, el estado "Conectado", los 3 documentos del KB y los modelos elegidos seguían todos ahí — y que Claude/Gemini/Llama aparecen deshabilitados con "próximamente" en el panel.

## Fase 9 — Carga automática del knowledge base, con upload de archivos como opción manual

El usuario pidió diseñar e implementar la forma "más consistente" para automatizar la obtención del KB, discutida como idea al cierre de la Fase 8, dejando además el upload de archivos como alternativa manual opcional.

### Por qué un endpoint en el probe del cliente, y no conectores a Notion/Drive/S3/etc.
Se evaluaron dos caminos: (a) que Vectora se integre directo con las fuentes del cliente (Notion, Google Drive, Confluence, S3...), o (b) que el propio proceso del cliente — que ya expone un servidor HTTP para el probe — también exponga su KB. Se eligió (b) porque reusa una pieza que ya existe y ya es de confianza (el mismo servidor que atiende `/probe/salud` y `/probe/ejecutar`), en vez de que Vectora tenga que guardar credenciales de N sistemas de terceros y mantener un conector por cada uno. El cliente decide de dónde saca esos documentos (su vector store, un export de Notion, archivos locales) — a Vectora solo le llega la lista ya resuelta.

### El diseño: `probe.exponerKb(docs)` + `GET /probe/kb`, opcional
- SDK (`probe/src/probe.ts`, `types.ts`): nuevo método `exponerKb(docs: KbDoc[])` que guarda los documentos en memoria y los sirve en una ruta nueva del servidor HTTP local, `GET /probe/kb`. Si nunca se llama, la ruta devuelve 404 con un mensaje explícito — nada se rompe para quien no lo use. `salud()` ahora incluye `tieneKb: boolean`, así el server de Vectora sabe si ofrecer el botón sin tener que golpear `/probe/kb` primero.
- Server: nueva ruta `POST /api/casos-de-uso/:id/obtener-kb` (mismo patrón que `verificar-conexion`: recibe `probeUrl`, hace `GET {probeUrl}/probe/kb` con timeout, devuelve los docs). `verificar-conexion` ahora también devuelve `tieneKb` (leído de `salud()`).
- Cliente: en el paso "Knowledge base" del stepper, si `tieneKb` vino en `true` al verificar la conexión, aparece un botón "Cargar automáticamente desde tu sistema" que llama a esa ruta y reemplaza `kbDocs` con lo que devolvió — cero copy/paste.
- **Upload de archivos, la opción manual pedida explícitamente:** un botón "Subir archivo(s) .md/.txt" que lee los archivos con `File.text()` en el navegador (sin tocar el server ni el probe del cliente) y los agrega a `kbDocs` — el título sale del nombre de archivo. Convive con la carga automática, la edición manual campo por campo, y "Usar KB de ejemplo": las cuatro formas terminan escribiendo al mismo estado `kbDocs`, así que ninguna es "la buena" — son formas distintas de llenar el mismo dato.
- `examples/cliente-demo` ahora llama `probe.exponerKb(todosLosDocumentos())` (`retrieval.ts` + `index.ts`), así que sirve de demo en vivo del camino automático, no solo de la conexión.

Se aprovechó para corregir dos comentarios desactualizados encontrados de paso en `docs/COMO-FUNCIONA-LA-CONEXION.md` § 4 (todavía describían `wrap` como "obligatorio en el punto donde se llama al modelo", un resabio de antes de la Fase 7) y en `probe/src/types.ts` (`ProbeOptions.apiKey` sugería que sin `apiKey` el cliente "sigue llamando a los modelos con su propia key vía wrap").

Verificado con Playwright en navegador real contra `cliente-demo` corriendo de verdad: el botón de carga automática trajo los 10 documentos reales del KB con un click, y el upload de un archivo `.md` de prueba cargó título (del nombre de archivo) y contenido correctamente.

## Fase 10 — Preparación para desplegar en vivo (Vercel + Supabase + Render), lo más barato posible

El usuario quiere probar un caso demo real, no local: el cliente (`client/`) en Vercel, la base de datos en Supabase, y una recomendación de stack priorizando costo mínimo.

### Por qué el server y `cliente-demo` NO van en Vercel (aunque el usuario solo conocía Vercel/Supabase)
Dos características del código, confirmadas leyendo el código real (no supuestas):
1. `orchestrator.ts::iniciarCorrida()` dispara la corrida completa **fire-and-forget** (`void ejecutarCorridaParaGobernanza(...)`, sin `await`) y devuelve el response de inmediato — una función serverless de Vercel se congela/mata apenas se manda la respuesta, así que la corrida de fondo (potencialmente minutos, ~150 llamadas a `/probe/ejecutar`) moriría a mitad de camino.
2. `probe.ts::levantarServidor()` abre un `http.Server` de Node (`createServer(...).listen(puerto)`) que queda escuchando indefinidamente — no hay "puerto" en un modelo serverless de función-por-request.

Por estas dos razones (no por preferencia), tanto el server de Vectora como `examples/cliente-demo` necesitan un host de proceso persistente. Se evaluaron Render, Railway y Fly.io — se eligió **Render** para la recomendación de "más barato posible" porque mantiene un tier free real para web services (con la salvedad de que se duermen tras inactividad, ver limitación abajo), a diferencia de Railway (ya no tiene free tier perpetuo, mínimo ~US$5/mes) y de la política cambiante de Fly.io.

### Cambio de código necesario: SQLite → Postgres, un solo datasource
Prisma no soporta dos providers en el mismo schema. Se le preguntó al usuario explícitamente cómo resolverlo (mantener SQLite local + cambiar solo para deploy, vs. Postgres para todo) — eligió **Postgres para todo**. Cambio real en `server/prisma/schema.prisma`: `provider = "sqlite"` → `"postgresql"` (la nota de diseño en el propio schema ya anticipaba esto — los campos "tipo" se guardan como `String` validados por zod, no como enums nativos, justamente para que esta migración no tocara nada más). Se regeneró el cliente de Prisma (`npx prisma generate`) y se confirmó que el server sigue compilando sin cambios de tipos. **Efecto colateral real:** el `dev.db` local queda inútil — hace falta un `DATABASE_URL` de Postgres real (Supabase) también para desarrollo local, no solo para producción. `server/.env.example` se actualizó para reflejar el formato de connection string de Postgres en vez de `file:./dev.db`.

### `render.yaml` (Blueprint) para desplegar los dos servicios de Render con un solo click
Se creó en la raíz del repo, con los build/start commands exactos verificados contra los `package.json` reales (no inventados): `server` necesita `npm run build -w @vectora/probe` antes que su propio build porque `@vectora/probe` se resuelve por su campo `main` (`./dist/index.js`) — ni `tsx` ni Node lo transpilan al vuelo al importarlo como dependencia. `cliente-demo-fintech` corre con `tsx` directo (sin build propio) pero también necesita `@vectora/probe` ya buildeado. El `startCommand` de `cliente-demo` fija `VECTORA_PROBE_PORT=$PORT` porque el SDK lee esa variable, no `PORT` — sin este mapeo, el proceso escucharía en el puerto default (4500) en vez del que Render le asigna.

### La limitación real para una demo en vivo: cold start del tier free de Render
Un web service free de Render se duerme tras ~15 min sin tráfico y tarda 30-60s en volver a levantar — más que el timeout de 10s de "verificar conexión" (`TIMEOUT_VERIFICACION_MS`). Para una demo en vivo, la recomendación es "despertarlo" con un curl minutos antes, o pagar el tier más barato "always-on" (Render Starter, ~US$7/mes por servicio) solo durante la ventana de la demo — Render cobra por hora/prorrateado, así que se puede subir de tier el día de la demo y volver a free después, sin pagar un mes completo por unas horas de uso.

## Fase 11 — El deploy de `vectora-server` en Render falló: `devDependencies` no se instalan en producción

El usuario reportó que el deploy falló, sin logs a mano (yo no tengo acceso al dashboard de Render). En vez de adivinar a ciegas, se reprodujo el problema localmente clonando el repo a una carpeta limpia y corriendo `NODE_ENV=production npm install` — la misma condición que usa Render — para ver qué se rompía de verdad, sin especular.

### La causa raíz: tres paquetes de build/runtime vivían en `devDependencies`
`NODE_ENV=production npm install` omite `devDependencies` (instaló 74-77 paquetes en vez de los 233 del árbol completo de dev). El repo tenía, en `probe/`, `server/` y `examples/cliente-demo`, paquetes que son imprescindibles para build o para runtime en producción pero estaban clasificados como solo-para-dev:
- `typescript` — sin él, `tsc` ni siquiera existe (`sh: tsc: command not found`, el primer error que se reprodujo).
- `@types/node` — sin él, `tsc` compila pero falla con `Cannot find name 'process'/'fetch'/'console'/'AbortController'` por todo `probe.ts`, `providerGateway.ts`, `casosDeUso.ts`, etc. — cualquier archivo que toque APIs globales de Node.
- `prisma` (el CLI, no `@prisma/client`) — sin él, no hay forma de generar el cliente de Prisma en el server nuevo.
- `tsx` en `examples/cliente-demo` — su `start` (`tsx src/index.ts`) no tiene build compilado propio en el flujo de Render; sin `tsx` instalado en runtime, el proceso ni arranca.

Se movieron los cuatro a `dependencies` en los `package.json` correspondientes. Además se agregó `"postinstall": "prisma generate --schema prisma/schema.prisma"` a `server/package.json` (para que el cliente de Prisma se regenere solo en cualquier instalación futura, no solo en Render) y se reforzó el `buildCommand` de `vectora-server` en `render.yaml` con un `npx prisma generate` explícito antes del build, como cinturón y tirantes.

### Verificado reproduciendo la falla real, no solo revisando el diff
Se clonó el repo a una carpeta temporal, se corrió `NODE_ENV=production npm install` seguido del `buildCommand` exacto de `render.yaml` — falló exactamente como se esperaba (`tsc: command not found`, después los errores de `@types/node` al arreglar eso). Con los cuatro paquetes movidos, se repitió el mismo experimento desde cero (clon limpio, `npm install` con `NODE_ENV=production`, build, y arranque del proceso compilado con un `DATABASE_URL` falso) — build limpio, y el server efectivamente levanta y responde HTTP; el único error que queda es el esperado (`Can't reach database server at localhost:5432`), que desaparece con un `DATABASE_URL` real de Supabase. Mismo experimento para `cliente-demo-fintech`: build y arranque con `tsx` limpios, `GET /probe/salud` responde `tieneKb:true`.

## Fase 12 — Supabase, IPv6, y el pooler se comió el seed

Migrar de local a Supabase real destapó dos problemas de infraestructura que no existían contra SQLite, ninguno de los dos era un bug de Vectora hasta que el segundo sí:

1. **`P1001: Can't reach database server` con la connection string directa de Supabase.** La conexión directa (`db.<ref>.supabase.co:5432`) resuelve solo por IPv6, y la red del usuario no tenía salida IPv6 — típico, no específico de este proyecto. Solución: usar el **connection pooler en modo sesión** de Supabase (`aws-0-<región>.pooler.supabase.com:5432`, usuario `postgres.<project-ref>`), que sí funciona por IPv4. Se documentó que hay que usar la misma URL del pooler tanto para correr `db push`/`db seed` en local como para el `DATABASE_URL` de `vectora-server` en Render, porque si la red del usuario no llega por IPv6, es razonable sospechar que Render tampoco (mismo síntoma, no confirmado pero consistente).

2. **Ese sí era un bug real de Vectora:** con el pooler ya andando, `npm run db:seed` murió con `FATAL: max clients reached in session mode - max clients are limited to pool_size: 15`. `seed.ts::sembrarCorreccionesJuicio` armaba 200 filas de `CorreccionJuicio` y las insertaba con `Promise.all(filas.map(f => db.correccionJuicio.create(...)))` — 200 conexiones simultáneas. Contra SQLite (sin pooling real) esto nunca había fallado; contra el pooler free de Supabase (tope de 15 conexiones concurrentes) revienta de inmediato. No era un problema de configuración de Supabase, era el seed pidiendo más concurrencia de la que cualquier Postgres con connection pooling razonable debería tolerar para una operación que es, en esencia, una sola escritura masiva.

   **Fix real, no un parche de configuración:** se cambió `Promise.all` de 200 `.create()` individuales por un solo `db.correccionJuicio.createMany({ data: filas })` — un único INSERT masivo, sin importar cuántas filas sean. Verificado con `tsc --noEmit` standalone contra el archivo (no se puede probar contra la DB real del usuario sin sus credenciales) — tipos correctos, mismos campos que el `.create()` original.
